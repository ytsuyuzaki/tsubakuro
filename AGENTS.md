# AGENTS.md

このファイルは、Codex、GitHub Copilot、Claude Code などの AI エージェントが、このリポジトリで作業するときに参照する共通ルールです。

## 適用範囲

- このファイルの内容は、リポジトリ全体に適用します。
- より下位のディレクトリに別の `AGENTS.md` がある場合は、そちらの内容を優先します。

## MCP 実装の参照先

- MCP エンドポイントや JSON-RPC メッセージ形式を変更する場合は、`mcp-adapter` プラグインの接続方式に準拠してください。

## テストコードの作成方針

- 機能追加や既存機能の挙動変更を行う場合は、原則としてテストコードを作成または更新してください。
- 作成したテストは、将来の機能変更時にリグレッションチェックとして活用できる内容にしてください。
- 表示文言、スタイル、レイアウトなど、見た目だけの変更では原則としてテスト追加は不要です。
- JavaScript などブラウザ上での挙動を変更する場合も、可能な限り自動テストを作成または更新してください。
- テスト追加が難しい場合は、理由と手動確認した内容を作業結果に明記してください。

## テストの選び方

- PHP のロジックや WordPress 連携の変更では、既存の PHPUnit テストを優先して追加・更新してください。
- ブラウザ操作、UI イベント、画面上の動作に関わる変更では、既存の E2E テストやブラウザテストを優先して追加・更新してください。
- 既存のテスト構成、命名、配置に合わせて、不要な新規フレームワークは追加しないでください。

## lint / test の実行

- コードや設定、テストを変更した場合は、変更内容に応じた lint や test を実際に実行して結果を確認してください。
- 実行したコマンドと結果は、作業結果として簡潔に共有してください。
- `.github/` ディレクトリ配下は、lint や test のチェック対象に含めなくて構いません。

## 変更時の最終確認ルール

- 機能追加・修正・リファクタリング・テスト修正を行った場合、作業完了前に `npm run lint` と `npm run test` を実行し、どちらも成功していることを確認してください。
- 変更範囲が限定的でも、最終確認は原則としてリポジトリ標準コマンド（`npm run lint` / `npm run test`）を優先してください。
- 実行環境の制約で全体実行が難しい場合は、理由を明記したうえで最小限の代替コマンド（例: `npm run lint:js`, `composer test`, `npx playwright test ...`）を実行し、未実施分を明確に共有してください。
- ドキュメントのみの変更など、コード挙動に影響しない変更ではテストを省略可能ですが、判断理由を作業結果に明記してください。

## フォーマットと静的チェックの基本ルール

- フォーマットのみで解決できる指摘は、手修正よりも既存ツール（`phpcbf`, `wp-scripts` 系）での修正を優先してください。
- JavaScript は ESLint エラーを残さないでください。特に `no-duplicate-imports` と `no-console` などの基本ルール違反を避けてください。
- PHP は WordPress Coding Standards（PHPCS）に従ってください。空白、改行、括弧位置、ファイル先頭コメント前の空行など基本書式違反を残さないでください。
- 文字列のエスケープ、入力値のサニタイズ、出力時のエスケープなど、WordPress プラグインとしての基本的な安全規約を維持してください。
